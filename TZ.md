# Техническое задание (ТЗ) на импорт пространств в Home Assistant

## 1. Описание проекта
[cite_start]**Цель:** Автоматизировать массовое создание пространств (Areas) в системе Home Assistant [cite: 76] [cite_start]в обход веб-интерфейса, используя Python-скрипт и WebSocket API[cite: 83, 84]. 

[cite_start]**Текущий статус:** Реализуется первый (подготовительный) этап, включающий чтение подготовленных файлов, трансформацию данных и их безопасный вывод в консоль без изменения конфигурации Home Assistant[cite: 115, 116].

---

## 2. Входные данные
[cite_start]Источником данных выступает предварительно обработанный файл `spaces.parquet`[cite: 96, 100]. 

[cite_start]Ключевые столбцы для работы[cite: 101, 102, 104]:
* `room_slug` — технический идентификатор комнаты в транслите (например, `402_kabinet_medits`).
* `space` — короткий идентификатор (например, `402_`).
* [cite_start]`floor` — номер этажа (использование отложено до следующих этапов)[cite: 114].

---

## 3. Требования к логике работы скрипта (Текущий этап)

1.  [cite_start]**Чтение данных:** Скрипт должен корректно открывать и считывать данные из `spaces.parquet` с помощью библиотеки `pandas`[cite: 107].
2.  [cite_start]**Формирование имени пространства (Name):** * За основу берется значение из столбца `room_slug`[cite: 113].
    * [cite_start]Скрипт должен преобразовывать технический транслит в удобочитаемое наименование на кириллице (например, `402_kabinet_medits` $\rightarrow$ `402_Кабинет медицинский`)[cite: 120].
    * [cite_start]Рекомендуемый метод: использование словаря соответствий (mapping)[cite: 120].
3.  **Формирование алиасов (Aliases):**
    * [cite_start]За основу берется значение из столбца `space`[cite: 114].
4.  **Режим тестирования (Dry Run):**
    * [cite_start]Скрипт должен поддерживать запуск в режиме "холостого прогона" (`--dry-run`)[cite: 116].
    * [cite_start]В этом режиме скрипт **не** подключается к Home Assistant, а только выводит в консоль сформированные JSON-запросы с ключом `type: "config/area_registry/create"`[cite: 93, 116].

---

## 4. Архитектурные и системные требования
Разработка ведется в строгом соответствии с файлом `RULES.md` и базовыми инструкциями:

* **Логирование:** Запрещено использование `print()`. [cite_start]Вывод информации (успешное чтение файлов, сгенерированные JSON) и ошибок осуществляется через модуль `logging`[cite: 119].
* **Безопасность:** Запрещено зашивать токены, пароли и IP-адреса в код. [cite_start]Они должны выноситься в переменные окружения (`.env`) или `config.yaml`[cite: 24]. 
* [cite_start]**Версионирование:** Разработка ведется с присвоением версий по системе SemVer (начиная с `v1.0.0`)[cite: 122]. Любые изменения фиксируются в Changelog.
* [cite_start]**Изоляция зависимостей:** Используемые библиотеки (в т.ч. `pandas`) фиксируются в `requirements.txt`[cite: 30].
